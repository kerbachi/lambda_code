# LLM Handler — Environment Compatibility Analysis

## What It Does

The LLM Handler is an **advisory** Lambda function that uses an LLM (AWS Nova Lite via LaunchPad Gateway) to analyze IAM Pull Requests for environment compatibility. It checks whether the target deployment environment (inferred from the repo name, branch, and PR title) matches the environment referenced in linked Jira tickets.

It runs as a **parallel check** alongside the three real validation checks (IAM Policy Compliance, IAM Boundary Requirement, AD Group Environment Check). It does **not** block PRs — its results are informational only.

## How It Works

### Data Flow

```
1. PR Poller detects new PR → writes to DynamoDB
2. DynamoDB Stream triggers stepfunction_trigger Lambda
3. stepfunction_trigger generates a UUID execution_id
4. stepfunction_trigger fetches PR data from GitHub, Jira tickets,
   and stores everything in S3 under {execution_id}/
5. stepfunction_trigger starts Step Function with execution_id
   injected into the event payload
6. Step Function invokes LLM Handler in parallel with other checks
7. LLM Handler reads PR data from S3 using execution_id (not GitHub directly)
8. LLM Handler builds a prompt, invokes the LLM via LaunchPad Gateway
9. LLM returns structured JSON analysis
10. Results Aggregator consolidates LLM result with validation check results
    and writes to DynamoDB
```

Note: S3 data is saved BEFORE the Step Function starts to avoid race conditions
where the LLM handler runs before the data is available.

### S3 Bucket Structure

Bucket name: `iam-automation-prs-and-jira-{account_id}`

```
{execution_id}/
├── pr_full_details.json    # Full GitHub PR API response
├── pr_files.json           # List of changed files with diffs
├── file_contents.json      # Actual file contents (base64 decoded)
└── {TICKET-ID}.json        # Jira ticket data (e.g., TTICID-12265.json)
```

The `execution_id` is a UUID generated by the stepfunction_trigger upfront
(before starting the Step Function). It is:
- Used as the Step Function execution name (so it matches the ARN suffix)
- Injected into the Step Function input at `$.dynamodb.NewImage.execution_id.S`
- Written to DynamoDB for tracking
- Used as the S3 prefix for all PR/Jira data

### LLM Invocation

The handler uses the **LaunchPad Gateway** (`cs_launchpad_ai` from Lambda layer CS-4423) as a proxy to AWS Bedrock. This is required by company policy for production use.

- Model: `aws-nova-lite-v1` (Amazon Nova Lite)
- The prompt is split into system + user messages to avoid LaunchPad guardrail blocks
- Retries up to 3 times with exponential backoff
- Timeout: 120s for the LaunchPad client, 300s for the Lambda itself

The `LLM_MODE` environment variable controls the invocation path:
- `launchpad` (default): Uses LaunchPad Gateway — required for production
- `direct`: Uses direct Bedrock API — only for dev testing (requires `bedrock:InvokeModel` in permissions boundary, which `AGT-IAM-BOUNDARY-BASE` does **not** include)

### Prompt Template

Located in `src/llm_handler/prompt_template.py` (version 3.1.0).

The system message tells the LLM to analyze PRs for environment compatibility and respond with JSON only. The user message includes:
- Repository name, PR number, title, author, branches
- PR description (truncated to 2000 chars)
- Changed files summary (up to 30 files)
- Jira ticket details (summary, status, priority, environment)

Expected JSON response fields:
- `pr_environment`: PRODUCTION / STAGING / DEVELOPMENT / TEST / UNKNOWN
- `jira_environment`: same values
- `environment_match`: MATCH / MISMATCH / UNKNOWN
- `confidence_score`: 0.0 to 1.0
- `overall_risk`: LOW / MEDIUM / HIGH
- `summary`, `security_risks`, `compliance_issues`, `recommendations`

### Jira Ticket Extraction

The handler loads Jira ticket JSONs from S3 (stored by stepfunction_trigger). If none are found, it falls back to regex extraction from the PR title and body using a list of known Jira prefixes (GCIT, GTBT, CS, TTICID, etc.).

### Empty Files Handling

If `pr_files` is empty but `pr_details.changed_files` indicates the PR has files, the handler logs a warning that the stepfunction_trigger may have failed to fetch files from GitHub. This helps distinguish "PR truly has no files" from "S3 data is missing or stale".

| Variable | Default | Description |
|---|---|---|
| `BEDROCK_MODEL_ID` | `aws-nova-lite-v1` | LLM model identifier |
| `BEDROCK_REGION` | `us-east-1` | AWS region for Bedrock |
| `MAX_TOKENS` | `4096` | Max output tokens |
| `TEMPERATURE` | `0.3` | LLM temperature |
| `ENVIRONMENT` | `dev` | Deployment environment |
| `PR_DATA_S3_BUCKET` | — | S3 bucket for PR data (`iam-automation-prs-and-jira-{account_id}`) |
| `LLM_MODE` | `launchpad` | `launchpad` or `direct` |
| `LAUNCHPAD_SECRET_NAME` | `{env}/launchpad-bedrock` | Secrets Manager name for LaunchPad credentials |
| `JIRA_BASE_URL` | — | Jira instance URL |


### LaunchPad Secret Structure

The secret at `{env}/launchpad-bedrock` contains:
```json
{
    "gateway_url": "https://...",
    "api_key": "...",
    "secret_key": "...",
    "app_id": "..."
}
```

## IAM Permissions

The LLM Handler Lambda role includes:
- **S3**: `s3:GetObject`, `s3:ListBucket` on `iam-automation-prs-and-jira-{account_id}`
- **Secrets Manager**: Read access to `{env}/launchpad-bedrock`, `{env}/jira*`, `github-enterprise-token`
- **Bedrock**: `bedrock:InvokeModel` on Nova Lite and Claude models (fallback path only)
- **VPC**: `AWSLambdaVPCAccessExecutionRole` (runs in VPC for LaunchPad Gateway access)

## Lambda Layer

Uses the `gts-launchpad-ai-example-{env}-layer` Lambda layer (CS-4423), which provides:
- `cs_launchpad_ai.LaunchPadClient` — HTTP client for the LaunchPad Gateway
- `cs_launchpad_ai.parse_lambda_response`, `extract_answer` — response parsing helpers

Layer ARN (dev): `arn:aws:lambda:us-east-1:286121356608:layer:gts-launchpad-ai-example-dev-layer:12`

## How to Modify

### Change the LLM Model

Update `BEDROCK_MODEL_ID` in `lambda.tf` under the `lambda_llm_handler` module's `environment_variables`.

### Change the Prompt

Edit `src/llm_handler/prompt_template.py`. The `build_environment_check_messages()` function returns the system and user messages. Update `PROMPT_VERSION` when making changes.

### Add Jira Prefixes

Add new prefixes to the `JIRA_PREFIXES` list in `src/llm_handler/lambda_function.py`. This list is used for fallback regex extraction when Jira ticket JSONs aren't available in S3. Also update the same list in `src/stepfunction_trigger/lambda_function.py` so the trigger fetches the tickets.

### Change LLM Invocation Mode

Set `LLM_MODE` to `direct` in the Lambda environment variables to bypass LaunchPad Gateway and call Bedrock directly. Only use this for dev testing — production must use `launchpad`.

## Results Aggregator

The **Results Aggregator** (`src/results_aggregator/lambda_function.py`) runs after the LLM Handler. It:
1. Reads the three validation check results from DynamoDB
2. Parses the LLM Handler output
3. Writes consolidated results (including `llm_analysis_endstatus`, `llm_analysis_details`, `llm_model_used`) back to DynamoDB

The LLM result is advisory — it doesn't change the overall pass/fail determination, which is based solely on the three real validation checks.
